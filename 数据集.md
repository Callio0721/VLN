## 3. R2R-CE数据集

R2R-CE 由 **VLN-CE**（Vision-and-Language Navigation in Continuous Environments）框架提出，主要对 R2R 进行了**连续空间导航（Continuous Navigation）**的改进，使得智能体能够在更现实的环境中进行移动。

下载路径

```bash
https://jacobkrantz.github.io/vlnce/data
```

[R2R_VLNCE_v1-3.zip](https://drive.google.com/file/d/1T9SjqZWyR2PCLSXYkFckfDeIs6Un0Rjm/view?usp=sharing) (3 MB)

**train (训练集)**: 10,819 个任务 (episodes)，61 个场景 (scenes)

**val_seen (可见验证集)**: 778 个任务，53 个场景

**val_unseen (不可见验证集)**: 1,839 个任务，11 个场景

**test (测试集)**: 3,408 个任务，18 个场景

[R2R_VLNCE_v1-3_preprocessed.zip](https://drive.google.com/file/d/1fo8F4NKgZDH-bPSdVU3cONAkt5EW-tyr/view?usp=sharing) (250 MB)

增加了一个数据增强划分 **`envdrop`**: 146,304 个任务，60 个场景（从 R2R-EnvDrop 项目移植而来）。

包含 **`embeddings.json.gz`** 文件，这是从 50 维的 GloVe 词向量文件中提取构建的。

指令的词汇标识符（Instruction tokens）已经被重新映射对齐，以匹配所提供的词向量。也就是说：在基础版中，`instruction_tokens` 里的数字（比如 `[1067, 159...]`）仅仅是根据当前数据集的频率排出来的简单索引。 但在构建预处理版时，作者为了让代码能直接对接现成的 50 维 GloVe 词向量字典，**把所有 `instruction_tokens` 里面的数字全部重新映射（重写）了一遍**。格式还是那个数组格式，但里面的整数 ID 全变了，必须和配套的 `embeddings.json.gz` 强绑定使用

**`{split}_gt.json.gz`** 包含了每个任务的底层真实动作（ground truth actions）和每一步的位置坐标（不包括 `test` 测试集）



### **文件一：`{split}.json.gz`（任务配置表/“考题”）**

这个文件定义了机器人的出生点、物理环境和语言指令。内部是一整个 JSON 字典，主要由两个键组成：`episodes` 和 `instruction_vocab`。

**数据格式示例：**

```json
{
  "episodes": [
    {
      "episode_id": 1, 
      "trajectory_id": 4,
      "scene_id": "mp3d/7y3sRwLe3Va/7y3sRwLe3Va.glb", // 对应的 Matterport3D 场景网格文件
      "instruction": {
        "instruction_text": "Go around the right side...", // 自然语言指令
        "instruction_tokens": [982, 141, 2202, 0, 0, 0] // 预处理后的词汇ID序列
      },
      "start_position": [-16.26, 0.15, 0.72], // 机器人在 3D 连续空间中的初始 XYZ 坐标
      "start_rotation": [0.0, 0.0007, 0.0, 0.999], // 机器人的初始朝向 (四元数)
      "goals": [
        {
          "position": [-12.33, 0.15, 4.21], // 终点坐标 (测试集对机器人不可见)
          "radius": 3.0 // 到达终点 3 米半径内算作成功
        }
      ],
      "reference_path": [ // 原始的连续参考路径点集合
        [-16.26, 0.15, 0.72], 
        [-16.28, 0.15, 2.41], 
        ...
      ],
      "info": {"geodesic_distance": 6.42} // 最短测地线距离等元数据
    }
  ],
  "instruction_vocab": { // 词汇表字典，用于将整数ID解码回单词
    "word_list": ["orchids", "order", "orient"], 
    "word2idx_dict": {"orchids": 1505, "order": 1506}, 
    "num_vocab": 2504,
    "UNK_INDEX": 1,
    "PAD_INDEX": 0
  }
}
```

### **文件二：`{split}_gt.json.gz`（底层动作标准答案）**

这是系统进行模仿学习时依赖的“参考答案”。字典的键是 `episode_id`（字符串格式）。

**数据格式示例：**

```json
{
  "1": { // 对应上方 episode_id 为 1 的任务
    "actions": [2, 2, 2, 1, 1, 0], // 标准答案动作序列 (例如：0=停止，1=左转，2=前进)
    "forward_steps": 27, // 整个轨迹中包含多少次“前进”动作
    "locations": [ // 每执行一个动作后，机器人所处的精确 XYZ 坐标
      [-16.267, 0.151, 0.720], 
      ...
      [-12.644, 0.151, 4.224]
    ]
  }
}
```

### 完整使用流程

当你启动代码库（如 `run.py`）去构建并训练一个视觉-语言导航代理时，这三个预处理文件是按照以下精密的工作流协同运转的：

#### 第一步：环境与任务初始化 (读取 `{split}.json.gz`)

1. **构建物理空间**：Habitat 物理引擎读取任务配置文件中的 `scene_id`，将对应的 3D 房屋网格（Mesh）加载到内存中。
2. **生成机器人实例**：将拥有物理碰撞体积的机器人放置在 `start_position` 指定的坐标上，并将其相机镜头对准 `start_rotation` 方向。
3. **分发考题**：系统抽取出该任务对应的 `instruction_tokens`（例如 `[1067, 159, 2389, ...]`）传给神经网络。

#### 第二步：语言特征编码 (读取 `embeddings.json.gz`)

1. **加载权重**：在构建模型的词嵌入层（`nn.Embedding`）时，系统直接读取这个 JSON 中的二维浮点数数组作为固定的初始权重。
2. **查表映射**：将传进来的整数 ID 序列 `[1067, 159, ...]` 瞬间转换成一个 `(序列长度, 50)` 的浮点数特征矩阵。
3. **序列记忆**：将这个矩阵送入双向 LSTM 网络，经过循环计算，提取出整句指令的上下文记忆状态。

#### 第三步：视觉感知与跨模态对齐 (物理引擎渲染)

1. **睁开眼睛**：机器人在当前的物理坐标处调用 Habitat 的传感器，渲染出当前第一人称视角的 RGB 彩色图像和 Depth 深度图像。
2. **提取视觉特征**：图像数据通过预训练的 ResNet 卷积神经网络，被压缩成高维的视觉语义特征。
3. **注意力聚焦**：策略网络（如跨模态注意力机制）将当前的“视觉特征”与上一步生成的“语言记忆状态”进行交叉比对，决定当前应该重点关注指令中的哪个词，以及画面中的哪个区域。

#### 第四步：预测与物理执行 (动作输出)

1. **输出决策**：网络最终输出一个概率分布，选出当前最应该执行的底层动作（例如：执行索引为 `1` 的 `TURN_LEFT` 动作）。
2. **物理交互**：将该控制信号发送给物理引擎，引擎计算碰撞，确保没撞墙后，将机器人的绝对坐标更新。

#### 第五步：强制纠正与误差回传 (读取 `{split}_gt.json.gz`)

1. **对答案**：在训练模式下，系统会立刻查阅该时间步在 `_gt` 文件中对应的“标准动作”到底是什么（例如标准答案是 `2`，即 `MOVE_FORWARD`）。
2. **计算误差**：发现模型预测了左转，但标准答案是直行。系统计算交叉熵损失（Cross-Entropy Loss）。
3. **进度监督**：同时提取 `_gt` 文件中的 `forward_steps`（总路径长度），计算当前机器人走完的比例，与模型的预测进度对比，计算均方误差。
4. **反向传播**：将所有误差累加，通过梯度下降算法更新整个导航网络的参数。

#### 第六步：循环迭代

重复第三步到第五步，机器人不断看画面、想指令、做动作、被纠正。直到网络主动输出“停止（STOP）”动作，或者步数达到上限，当前 Episode 结束，系统开始加载下一个场景数据。
